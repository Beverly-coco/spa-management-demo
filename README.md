# 方壶 SPA 管理系统（小程序端 + 管理后台）

## 项目概述

本项目是一套专为 SPA（水疗）店铺设计的综合性管理系统，旨在解决连锁店模式下库存管理、采购流程、以及日常运营中的诸多痛点。系统包含两大部分：

1.  **微信小程序端**：供门店员工使用，主要功能包括库存盘点、采购申请、月度用量统计等。
2.  **Web 管理后台**：供总公司管理员使用，用于管理商品、供应商、门店、审批采购单、以及查看各类数据报表。

该项目采用前后端分离的架构，前端技术栈涵盖了原生微信小程序开发和基于 React 的现代 Web 应用开发，后端则基于 Node.js 和微信云托管，实现了稳定、高效、可扩展的业务支持。

## 前端技术栈

### 1. 微信小程序端

- **核心框架**：原生微信小程序框架，未使用第三方 UI 库，保证了最佳的性能和灵活性。
- **状态管理**：通过 `app.js` 的 `globalData` 和页面的 `data` 实现简单的全局和页面级状态管理，符合小程序轻量化的特点。
- **组件化**：将可复用的 UI 和逻辑封装成自定义组件，提高了代码的复用性和可维护性。
- **网络请求**：封装了 `wx.cloud.callContainer`，通过统一的 `callCloudApi` 函数与云托管后服务进行交互，并对请求和响应数据进行了预处理和格式化。
- **UI/UX 设计**：
  - 自定义了 `tabBar` 和导航栏样式，品牌色彩统一，视觉体验良好。
  - 图标和图片资源管理清晰，并实现了图片懒加载和默认图处理，优化了用户体验。
  - 页面布局简洁明了，操作流程直观，符合移动端用户的使用习惯。

### 2. Web 管理后台

- **核心框架**：`React 18` + `React Router 6`，构建了现代化的单页面应用（SPA）。
- **UI 组件库**：`Ant Design 5` + `@ant-design/pro-components`，快速搭建出专业、美观、功能丰富的后台管理界面。
- **状态管理**：利用 `React Context` 和 `useState`、`useEffect` 等 Hooks 进行组件级和应用级的状态管理，实现了对用户认证状态的全局共享。
- **图表展示**：使用 `Chart.js` 和 `react-chartjs-2` 将复杂的报表数据可视化，如图表、曲线图等，使数据一目了然。
- **网络请求**：使用 `axios` 进行 HTTP 请求，并设置了请求拦截器，统一处理 API 请求和响应，特别是对 `401` 未授权状态进行了全局处理，自动弹出登录过期提示。
- **代码构建**：基于 `Create React App` (`react-scripts`) 进行项目构建和打包，支持热更新和生产环境优化。

## 项目亮点与技术难点

### 亮点

1.  **前后端分离与云原生架构**：项目采用前后端分离模式，小程序端和管理后台作为两个独立的前端应用，通过 API 与后端服务通信。后端部署在微信云托管上，利用其弹性伸缩、免运维的优势，降低了服务器成本和维护复杂度。
2.  **统一的认证与授权机制**：通过 JWT（JSON Web Token）实现小程序端和管理后台的统一用户认证。前端通过本地存储（`localStorage` / `wx.getStorageSync`）持久化 token，并通过请求头将 token 发送至后端进行验证。后端通过中间件统一校验 token 的有效性，实现了对 API 接口的权限控制。
3.  **优雅的会话管理**：在管理后台中，通过 `axios` 拦截器和全局事件监听，实现了对登录状态过期的优雅处理。当 API 返回 401 状态码时，会全局捕获并弹出模态框提示用户重新登录，提升了用户体验。
4.  **服务预热与性能优化**：在小程序启动时，通过 `warmupCloudService` 函数主动调用云服务的健康检查接口，触发云服务的“冷启动”过程。这有效避免了用户首次进行业务操作时因服务启动慢而导致的请求超时问题，显著提升了首次交互的响应速度。
5.  **响应式与多端适配**：管理后台基于 Ant Design 的响应式布局，能够良好地适配不同尺寸的桌面浏览器。小程序端则天然适配移动设备，保证了在各种手机上的体验一致性。

### 技术难点与解决方案

1.  **问题：微信小程序云托管冷启动延迟**

    - **描述**：微信云托管的服务在长时间无访问后会进入“休眠”状态。当用户首次发起请求时，需要经历一个“冷启动”过程，耗时可能长达数秒，极易导致 API 请求超时，用户体验差。
    - **解决方案**：在小程序 `onLaunch`生命周期中，设计并实现了一套“服务预热”机制。即在应用启动时，立即异步发起一个对后端健康检查接口（如 `/health` 或 `/ping`）的“哑”请求。这个请求的目的不在于获取业务数据，而在于“唤醒”后端服务。同时，为了增加成功率，还设计了带有延迟的重试机制（`retryWarmup`），在首次预热失败后，会进行 1-2 次的静默重试。通过这种方式，将服务启动的等待时间前置到应用加载阶段，确保了用户在执行实际业务操作时能够获得毫秒级的响应。

2.  **问题：前后端数据类型一致性，特别是金额处理**

    - **描述**：在前后端数据交互中，金额等高精度数字在 JSON 传输过程中可能会丢失精度或被错误地解析为字符串。这在涉及财务计算的场景中是致命的。
    - **解决方案**：在小程序端的 `callCloudApi` 封装函数中，增加了 `processDataForApi` 和 `processApiResponse` 两个辅助函数。`processDataForApi` 在发送请求前，会遍历数据对象，将所有符合金额格式的字段（如 `price`, `amount`）显式地转换为数字类型。`processApiResponse` 在接收到后端响应后，同样会进行一次数据清洗，确保前端拿到的金额数据类型正确。这套机制保证了数据在整个链路中的一致性和准确性。

3.  **问题：管理后台的复杂表单和数据联动**
    - **描述**：在商品管理、采购单创建等页面，存在大量表单项，并且部分表单项之间存在联动关系（如选择商品后，自动填充规格和单位）。
    - **解决方案**：充分利用 `Ant Design` 的 `Form` 组件和 `@ant-design/pro-components` 提供的能力。通过 `Form.Item` 的 `dependencies` 属性和 `onValuesChange` 回调，可以轻松实现表单项之间的依赖和动态更新。对于复杂的表单，将其拆分为独立的子组件，通过 props 和回调函数与父组件通信，保持了代码的清晰和可维护性。

## 如何运行

### 小程序端

1.  在微信开发者工具中导入 `miniprogram` 目录。
2.  修改 `app.js` 中的 `globalData.cloudEnv` 为您自己的云环境 ID。
3.  确保已开通微信云托管，并已部署后端服务。
4.  点击“编译”即可在模拟器或真机上预览。

### 管理后台

1.  进入 `wxcloud/admin-src` 目录。
2.  运行 `npm install` 或 `yarn` 安装依赖。
3.  修改 `src/services/api.js` (或其他 API 配置文件) 中的后端服务地址。
4.  运行 `npm start` 启动本地开发服务器。
5.  在浏览器中打开 `http://localhost:3000`。
